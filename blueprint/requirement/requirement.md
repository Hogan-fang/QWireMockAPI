# QWireMockAPI Requirement

## 1. Order Server 需求

模拟订单的支付处理、订单查询，以及在订单状态发生变化时回调商户系统。

---

## 2. Order 接口需求

### 2.1 请求内容
商户提供以下基本信息：
- 商户代码
- 订单参考号
- 金额与币种
- 简单商品清单

### 2.2 处理与落库规则
- 进入业务处理的订单，无论成功或失败，均记录到数据库。
- 接口返回主要订单请求信息、服务器生成的订单号以及处理结果。
- 同一商户下，订单参考号必须唯一；不满足唯一性时，拒绝处理。
- 无法被系统处理的订单请求，不记录数据库，并通过 HTTP 错误码和错误信息返回。

---

## 3. Query 接口需求

### 3.1 查询条件
- 使用商户侧订单参考号进行查询。

### 3.2 业务行为
1. 卡号规则：
   - 以 `4` 开头：返回 `400`，错误为“卡片无效”。
   - 以 `5` 开头：返回 `400`，错误为“余额不足”（业务错误）。
2. 结果映射规则：
   - 成功：订单状态为 `SUCCESS`，商品状态为 `PROCESSING`。
   - 失败：订单状态为 `FAIL`，并返回 `failReason`。
   - 成功场景下，不应返回空的错误信息字段。
3. 状态推进规则：
   - 30 秒后：`SHIPPED`
   - 60 秒后：`DELIVERED`
   - 当全部商品均为 `DELIVERED`：订单状态为 `COMPLETED`
4. 当状态发生变化时，触发回调。
5. 数据库持久化要求：
   - 卡号需掩码存储。
   - 不可存储 `CVV` / `expiry`。
6. 数据脱敏范围：
   - 查询结果、回调数据、接口返回数据中的卡号都必须掩码。
   - 所有返回与回调中均不得包含 `CVV` / `expiry`。

---

## 4. Callback Server 需求

模拟商户系统，接受 Order Server 的回调信息。返回使用 HTTP Code。

### 4.1 回调接收接口
- `POST`：
  - 校验通过：写入内存并返回 `200`。
  - 校验失败：返回 `400`。
- `GET`：
  - `UUID` 格式错误：返回 `422`。
  - 记录不存在：返回 `404`。
   - 记录存在：返回 `200`，数据结构需符合 `OrderResponse`。

### 4.2 存储与日志
- 仅使用内存存储并输出日志。
